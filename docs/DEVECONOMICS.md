# DevEconomics — cost & velocity of codeless development

> **What this is.** A living ledger of what it *costs* to build DigitalSecretary by asking Claude:
> where the work runs (LLM tokens vs local compute vs network), how much code comes out, and what that
> implies for speed and money. Claude maintains it — append a **per-feature entry** after each change
> (template at the bottom). Pair it with `docs/DEVELOPMENT_JOURNAL.md` (what/why) and
> `docs/QUALITY_REPORT.md` (build outcome).
>
> **Ground truth for tokens/cost is `/cost`.** Only the Claude Code harness knows exact token usage for
> a session. Numbers here that aren't `measured` are `estimate` — always reconcile against `/cost`
> (this session) or the Anthropic Console usage dashboard (billing period).

---

## The three cost buckets

Every activity in a session lands in exactly one bucket. Only the first costs money per use.

| Bucket | Runs on | Costs | Examples |
|--------|---------|-------|----------|
| **A · LLM tokens** | Anthropic (Claude) | **$ per token** (the only metered cost) | Reading code to understand it, planning, generating code/tests/docs, editing, deciding |
| **B · Local compute** | your PC | $0 tokens (electricity/time only) | `dotnet build`/`test`, `build.ps1 -All`, the Python doc generators + consistency checker + secret scan, `DocShots` screenshots, manual build, `git` + pre-commit hook |
| **C · Remote, no tokens** | a network service | $0 tokens | NuGet restore from nuget.org (MailKit, Google.Apis.\*). *No* cloud LLM agents or `/code-review ultra` were used in this work. |

**The core economic fact:** the entire **verification and documentation apparatus is Bucket B** — deterministic
local scripts that cost zero tokens. Tokens are spent almost entirely on **understanding** the codebase and
**authoring** new content. That is the lever: anything you can move from A to B gets cheaper and more
repeatable.

## Rate card (per 1M tokens, Opus 4.8 — this session's model)

| | Input | Output | Cache read (0.1×) | Cache write (1.25×, 5‑min) |
|---|------:|-------:|------------------:|---------------------------:|
| **Opus 4.8** (`claude-opus-4-8`) | $5.00 | $25.00 | ~$0.50 | ~$6.25 |
| Sonnet 4.6 (`claude-sonnet-4-6`) | $3.00 | $15.00 | ~$0.30 | ~$3.75 |
| Haiku 4.5 (`claude-haiku-4-5`) | $1.00 | $5.00 | ~$0.10 | ~$1.25 |

_Prompt caching: reads ≈ 0.1× input, writes ≈ 1.25× (5‑min TTL) or 2× (1‑hour). The long, stable prefix
(system prompt, tool list, already-read files) is cached, so most re-sent context bills at the read rate —
caching is the single biggest cost reducer in a long agentic session._

## Why this is already cheap — and how to make it cheaper

The economics work because **~95% of the activity list is Bucket B/C (zero token)** and the token spend is
concentrated in two places: *understanding* and *authoring*. Attack those.

1. **Scaffold to move authoring A→B.** A `tools/new-feature.ps1 -Name X -Category Y` generator that stamps
   the boilerplate (`.csproj`, `plugin.json`, `…Module.cs`, a control skeleton, `FEATURE.md`, a test
   skeleton, and the requirement/user-guide doc stubs) would convert the most repetitive ~40–50% of
   generated lines into deterministic local codegen. Claude then writes only the feature-specific logic.
   **Biggest single win** — directly cuts output tokens, the priciest bucket.
2. **Shrink exploration with a one-page "feature recipe."** Most input tokens this session went to
   re-reading 15+ files to re-derive the pattern. `docs/ADDING_A_FEATURE.md` already helps; keeping a tight
   "patterns + gotchas" cheat-sheet (and the `coverage.runsettings` exclusion rule, the `check_docs` ID
   regex, the manual anchor/screenshot contract) means the next feature reads ~3 files, not ~15.
3. **Reduce duplication with a shared (non-contract) helper lib.** Gmail duplicated EmailDownloader's
   IMAP/naming/scanner because features can't reference each other. A `DigitalSecretary.FeatureKit`
   (referenced by features, separate from the tiny `Abstractions` contract) for file-naming, attachment
   scanning, and a generic IMAP core would cut both LOC and the tokens spent regenerating near-identical
   code — at the cost of a shared dependency. Worth it once a third mail-like feature appears.
4. **Right-size the model per phase.** Design/architecture decisions justify Opus; mechanical boilerplate
   and doc stubs could run on Sonnet 4.6 (0.6× the rate) or Haiku 4.5 (0.2×). Reserve Opus for the parts
   where capability changes the outcome.
5. **Keep caching warm.** The long stable prefix is what makes re-sent context bill at 0.1×. Avoid
   churning the system prompt / tool set mid-task; batch independent file reads into one turn (already done
   here) so fewer round-trips each re-pay setup.
6. **Keep verification in Bucket B, never ask the LLM to "review" what a script can check.** The gate
   (`build.ps1 -All`), `check_docs.py`, and `check_secrets.py` already do this — every check that becomes a
   deterministic script is a check that never costs a token again. Extend the pattern (e.g. generate doc
   stubs from the docgen data tables) rather than hand-writing+hand-reviewing.
7. **Generate docs/traceability from data, not prose.** The spreadsheets are already generated from Python
   data tables (Bucket B). Extending that to emit the requirement/user-guide `.md` skeletons would remove
   another chunk of token-authored text.

**Rule of thumb:** every artifact you can generate or verify with a deterministic script is a permanent
move from Bucket A to Bucket B. That is what makes "develop by asking" economically viable at scale.
